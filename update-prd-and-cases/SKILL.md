# 根据测试结果更新 PRD 与测试用例

## 背景

每轮测试结束后，测试产物会暴露：
- PRD 未描述清楚或遗漏的业务规则
- 现有测试用例未覆盖到的场景

这个 skill 的目的是把这些发现系统性地回写到 PRD 和测试用例中，确保文档与实际业务保持同步。

## 测试用例管理策略

### 双产物策略

| 产物 | 用途 | 管理方式 |
|------|------|----------|
| **测试用例总表** | 沉淀完整测试资产，供后续回归测试使用 | 持续维护，只增不减，分类管理 |
| **本轮增量导入 Excel** | 本轮新增/修正后需要导入云效的内容 | 每轮单独生成，导入后归档 |

### 总表 vs 增量表的价值

- **总表**：不是没用，它是完整测试资产的沉淀。每次回归测试都基于总表，而不是每次从零开始导入。
- **增量表**：解决"每次都整表重导"的问题。只导入本轮新增/修正项，云效侧可选择"追加"模式。

## 工作流程

### Phase 1: 定位最新测试轮次

1. 读取 `agent-context/testing/` 目录，找到编号最大的 `round-N/` 目录
2. 读取该轮次下的所有产物：
   - **缺陷清单**（`.xlsx`）— 用 pandas 读取，提取标题、优先级、状态
   - **改进建议**（`improvements.md`）— 直接读取
   - **测试报告**（`.pdf`）— 尝试用 PDF skill 读取，若不可用则跳过
3. 读取 `agent-context/testing/README.md` 了解目录规范

### Phase 2: 读取现有文档

1. **PRD**：读取 `agent-context/prototype/**/docs/PRD.md`，或 `agent-context/` 下的 PDF 版
2. **测试用例总表**：找到 `agent-context/` 根目录下的 `.xlsx` 测试用例文件，用 pandas 读取
3. 记录：
   - 总表的列结构（表头字段）
   - 现有用例的目录分布（哪些模块已有用例、哪些是空的）
   - 现有用例数量

### Phase 3: 分析差距

#### PRD 差距
逐条检查缺陷和改进建议，识别以下类型的补充项：
- 业务规则遗漏（如定位必须用腾讯地图经纬度、库存默认充足）
- 真实数据缺失（企业名称、地址、业务范围）
- 交互约束（模拟器适配、页面返回方式）
- 后台组织逻辑（系统核心架构思想）

#### 测试用例差距
对比缺陷/改进建议与现有用例：
- 识别**全新场景** — 现有用例完全未覆盖
- 识别**部分覆盖** — 现有用例有类似场景但步骤/预期不完整
- 区分来源：标记每条对应的是缺陷#N 还是改进建议条目

### Phase 4: 与用户对齐（必须）

**在动手修改之前**，将分析结果整理成表格呈现给用户：

```
## PRD 补充项
| 摘要 | 来源 | 建议补充位置 |
|------|------|-------------|
| xxx  | 缺陷#1 | 功能模块-业务规则 |

## 测试用例总表新增/修正
| 用例名 | 来源 | 建议放在哪个模块 |
|--------|------|-----------------|
| xxx    | 改进建议#2 | 订单管理 |

## 本轮增量导入 Excel 应收录
| 用例名 | 来源 | 变动类型 |
|--------|------|----------|
| xxx    | 缺陷#3 | 新增 |
| yyy    | 改进建议#1 | 修正 |
```

**明确告知用户**：
- PRD 直接更新到最新版本（覆盖式，非增量）
- 总表会补充新用例到合适模块（不是机械堆到末尾）
- 增量表只收录本轮需要导入云效的部分

等用户确认后再进入 Phase 5。

### Phase 5: 执行更新

#### 5.1 更新 PRD

1. 读取现有 PRD
2. 直接在对应章节修改/补充，保持结构清晰
3. 不做增量追加，而是更新到最新版本

#### 5.2 维护测试用例总表

1. 用 openpyxl 加载总表 xlsx（`load_workbook`，不用 `data_only=True`）
2. 分析现有模块分布，把新用例放到合适模块下（不是无脑追加末尾）
3. 如果某个模块已有用例，在该模块最后一行后追加
4. 如果是修正现有用例，更新对应行
5. 保存总表

#### 5.3 生成增量导入 Excel

1. 创建新 Excel，复制总表的表头
2. 只写入本轮新增/修正的用例（从 Phase 4 确认的列表）
3. 添加"变动类型"列，标记"新增"或"修正"
4. 添加"来源"列，追溯到缺陷#N 或改进建议条目
5. 命名规范：`测试用例-第N轮增量导入-YYYY-MM-DD.xlsx`
6. 放到 `agent-context/testing/round-N/` 目录

#### 5.4 归档

- 增量 Excel 生成后，移动到 round-N 目录
- 更新 testing/README.md，记录本轮新增的增量文件

### Phase 6: 输出报告

完成后向用户报告：

```
## 本轮测试资料摘要
- 缺陷清单：N 条
- 改进建议：M 条
- 测试报告：已读取 / 无法读取

## PRD 更新
- 补充了 xxx 业务规则
- 修正了 xxx 描述

## 测试用例总表
- 新增用例：X 条
- 修正用例：Y 条
- 分布模块：订单管理(Z条)、商品管理(A条)...

## 本轮增量导入 Excel
- 文件名：测试用例-第N轮增量导入-YYYY-MM-DD.xlsx
- 包含用例：X 条（新增 Y 条，修正 Z 条）
- 已放置到 agent-context/testing/round-N/

## 可追溯记录
| 用例名 | 变动类型 | 来源 |
|--------|----------|------|
| xxx    | 新增     | 缺陷#1 |
| yyy    | 修正     | 改进建议#2 |
```

## 关键约束

- **先对齐再动手**：Phase 4 必须等用户确认，否则不执行 Phase 5
- **PRD 覆盖式更新**：不是增量追加，是更新到最新版本
- **总表分类管理**：新用例放到合适模块，不是无脑堆末尾
- **增量表独立生成**：每轮单独生成，用于云效导入
- **全量可追溯**：每条改动都必须能追溯到具体缺陷编号或改进建议条目
- **遵守 AGENTS.md 优先级**：测试用例 > 测试报告 > 缺陷清单 > 改进建议 > 原型 > PRD
