# 根据测试结果更新 PRD 与补充测试用例

## 背景

每轮测试结束后，测试产物（缺陷清单、改进建议、测试报告）中会暴露出 PRD 未描述清楚或遗漏的业务真相，
以及现有测试用例未覆盖到的场景。这个 skill 的目的是把这些发现系统性地回写到 PRD 和测试用例中，
确保文档与实际业务保持同步。

## 工作流程

### Phase 1: 定位最新测试轮次

1. 读取 `agent-context/testing/` 目录，找到编号最大的 `round-N/` 目录
2. 读取该轮次下的所有产物：
   - **缺陷清单**（`.xlsx`）— 用 pandas 读取，提取标题、优先级、状态
   - **改进建议**（`improvements.md`）— 直接读取
   - **测试报告**（`.pdf`）— 尝试读取，若 PDF 工具不可用则跳过并告知用户
3. 读取 `agent-context/testing/README.md` 了解目录规范

### Phase 2: 读取现有文档

1. 读取当前 PRD — 优先找 `agent-context/prototype/**/docs/PRD.md`，如果不存在就找 `agent-context/` 下的 PDF 版
2. 读取当前测试用例 — 找 `agent-context/` 根目录下的 `.xlsx` 测试用例文件
3. 用 pandas 读取测试用例 xlsx，记录现有列结构和所有用例标题（用于去重）

### Phase 3: 分析差距

对比测试产物和现有文档，识别两类差距：

#### PRD 差距
逐条检查缺陷和改进建议，提取出以下类型的 PRD 补充项：
- **业务规则遗漏** — 测试暴露了 PRD 没说清的规则（如定位必须用腾讯地图经纬度、库存默认充足）
- **真实数据缺失** — 实际企业名称、地址、业务范围等 PRD 未记录
- **交互约束** — 模拟器适配、页面返回方式等 PRD 未提及的约束
- **后台组织逻辑** — 系统核心架构思想（如商家→商品的主线）PRD 未明确

#### 测试用例差距
逐条检查缺陷和改进建议，与现有用例标题比对，识别**全新场景**：
- 如果某个缺陷对应的验证场景在现有用例中已存在（标题或步骤描述能覆盖），则**不新增**
- 只新增现有用例完全未覆盖的验证场景
- 区分来源：标记每条新增用例对应的是缺陷还是改进建议

### Phase 4: 与用户对齐

**在动手修改之前**，将分析结果整理成表格呈现给用户：

1. PRD 补充项列表 — 每项标明内容摘要和来源（缺陷#N / 改进项）
2. 新增测试用例列表 — 每条标明用例名、对应问题来源、为什么现有用例没覆盖

明确告知用户：
- 只补充新增内容，不重写已有内容
- 现有测试用例保持不动，只在末尾追加

等用户确认或调整后再进入 Phase 5。

### Phase 5: 执行更新

#### 更新 PRD
1. 读取现有 PRD.md
2. 在对应章节追加新内容，保持原有结构不变
3. 如果原有结构太简略无法追加，可以扩展章节，但不删除原有内容
4. 新增内容用清晰的小节标题组织

#### 追加测试用例
1. 用 openpyxl 加载现有 xlsx（`load_workbook`，不用 `data_only=True`）
2. 读取表头列顺序，确保新行严格匹配现有列结构
3. 在最后一行之后追加新用例行
4. 填写字段：标题、目录、负责人（复用现有值）、前置条件、步骤描述、预期结果、优先级
5. 保存文件

#### 整理文档（如用户要求）
- 检查 PRD 是否有重复或矛盾的描述，合并整理
- 检查测试用例目录（第二列）分类是否合理，如有需要建议调整

### Phase 6: 输出报告

完成后向用户报告：
- PRD 新增了哪些内容（简要列表）
- 测试用例新增了几条、分别对应什么问题
- 如果有整理调整，说明改了什么

## 关键约束

- **不删除现有内容** — 只追加和扩展，不删改已有 PRD 描述和测试用例
- **先对齐再动手** — Phase 4 必须等用户确认
- **保持 xlsx 格式一致** — 新增行的列顺序、字段命名必须与现有行完全一致
- **标注来源** — 每条新增项都要能追溯到具体缺陷编号或改进建议条目
- **遵守 AGENTS.md 优先级** — 测试用例 > 测试报告 > 缺陷清单 > 改进建议 > 原型 > PRD
